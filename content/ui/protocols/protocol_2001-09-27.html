2. Interface of replaceable context menus

We discussed several designs to provide replaceable context menus. We found at least two possible solutions. The first one would pass the context as a set of affectect objects, which would give a similar behaviour as for the own context menus. The second solution would be to only pass the position and the window. So the context menu itself must collect all necessary informations. Additionally the new context menu might also call back to ask for more details on the given position and window.

Because we couldn't assess however exactly which requests are needed for such an interface, we have adjourned a decision about this.

3. Replaceable Accelerators

After some short discussions about the 'usefullness' and the effort for such a feature, we came to the conclusion, that it would be to expensive and that we don't need it at all.
